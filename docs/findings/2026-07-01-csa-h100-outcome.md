# Finding 2026-07-01 — CSA learning-curve sweep on H100: implementation validated, sweep compute-bound (honest outcome)

Record of the autonomous session's attempt to run the CSA learning-curve sweep on a rented H100 and
gather results for the preprint. **Outcome: the batched-parallel CUDA cognate-matcher was built and
validated; the full sweep was not completed** (H100 instance instability + workload is
orchestration-bound). Reported honestly rather than asserted.

## What was built + validated (committed, real deliverables)

- **`scripts/baselines/csa_batched.py`** — a batched-parallel CUDA energy for Tamburini's CSA. His
  `CoupledAnnealer` CUDA path is stuck in slow serial (the 16-annealer `mp.Pool` forks CUDA tensors
  and dies), firing `16 × N_lost` tiny `editdistance1N` kernels per step. `csa_batched` stacks every
  rewritten query word across all 16 annealers into ONE tiled pairwise `editdistance` kernel per
  step, reusing his exact `State2Assignment` / `expandWord` / `lsa_2g` / penalties.
  - **Validated bit-identical to serial on CPU**: tiled-pairwise == `editdistance1N` (max diff 0.0);
    `batch_energy([s...]) == [energy(s)...]`; full annealing trajectory serial == batched
    (best_energy 60.169151 == 60.169151, identical state). Key subtlety fixed: his `move()` mutates
    state in place and 16 annealers share one init object → each probe is snapshotted before the next
    move.
  - **GPU memory-leak fixed**: `torch.no_grad()` around the edit-distance ops (without it an autograd
    graph accumulated every step → silent OOM-kill by ~step 100–300; the fix ran 500 steps flat at
    1.2 GB).
- **`scripts/baselines/run_tamburini.py`** — threaded a `device` param (default `cpu`; `cuda`
  engages his CUDA-capable `EditDistanceWild`) + an `on_checkpoint` plateau early-stop + a `batched`
  hook, all default-off so the validated CPU reproduction and the 362-test suite are unchanged.
- **`scripts/baselines/csa_sweep.py`** — H100 driver: checkpoint/resume per cell, plateau early-stop,
  preflight, small-sizes-first ordering. **`scripts/baselines/csa_local_curve.py`** — CPU runner.
- The deterministic half (downsample, chance floor, Linear-A locator) was already in
  `learning_curves.py`. Full suite after all changes: **362 passed, 4 xfailed.**

## Why the sweep did not complete

1. **The rented H100 instance was unstable for this Python workload.** Individual cells ran fine
   through the driver (`CELL1_OK`/`CELL2_OK`/`LINB_OK`, GPU active), but every attempt to launch the
   *full* multi-cell sweep as a background process stalled at startup with the GPU never engaging
   (CUDA context never initialised), across `tmux`, `setsid`, and `nohup`. The instance was destroyed
   after exhaustive attempts to stop billing.
2. **The workload is orchestration-bound, not compute-bound.** Even batched, the full-size primary
   benchmark (linearb-greek, 919 words) is ~27 s/step; the GPU is starved (nvidia-smi showed ~1 %
   time-averaged util) because the per-step work is dominated by Python/transfer overhead, not the
   edit-distance kernel. This is *itself a finding*: an H100 is the wrong tool for Tamburini's CSA as
   written — the speedup that matters is CPU process-level fan-out / batching, not raw GPU FLOPs.
3. Local CPU runs were contended by orphaned pool workers from earlier `processes=16` launches.

## How it reads in the paper

§8.3–§8.4 report the honest state — **implemented-and-validated, not-yet-swept** — and ship the
runnable harness + chance floor + Linear-A locator so the accuracy-vs-size curve is *reproducible and
auditable* rather than asserted. The paper's argument does not depend on the curve: it is a
supporting information-sufficiency diagnostic, and the core (discipline machinery + the pre-registered
nulls) stands independently.

## Update — the sweep WAS exercised end-to-end (reduced budget); a diagnostic lesson

After destroying the H100 I ran the sweep on CPU via the validated batched matcher and got **6 real
measured points** (`results/csa/curve_lowerbounds.json`): luvian-Hittite sizes 20–59 and linb sizes
100–200, all at/within noise of the analytic chance floor. These are **under-converged lower bounds**
(300–500 steps; CSA converges near ~4,000) and the per-step CPU cost is ~quadratic in lexicon size
(linb sz100 ≈ 4 min, sz200 ≈ 17 min, sz900 would be hours), so the *converged* full-range curve is
hours-to-days of compute. §8.3 reports the honest state: **implemented, validated, and exercised —
but not converged.**

**Diagnostic lesson (recorded so I don't repeat it):** for much of this session I misread bursty
instantaneous `%CPU`/GPU-util snapshots (0.0 % between compute bursts) as "hung" and repeatedly
relaunched — including a probably-premature H100 teardown. The processes were computing the whole
time (accumulated CPU **time** was advancing); cells just take 1–17 min each. Always check the
**delta in accumulated CPU time** (or cell files landing), never a single `%CPU` sample.
